# 经济系统

> **Status**: Designed
> **Author**: user + agents
> **Last Updated**: 2026-04-03
> **Implements Pillar**: 支柱3（联动 > 单体——资源分配驱动构筑选择）

## Overview

经济系统管理玩家的代币（Token）资源。代币是游戏内唯一货币，通过击败敌人获得，
在商店中花费购买出千能力。经济系统追踪余额、处理收支、判断购买可行性。

设计目标：45 代币（全通关）购买 7-8 个道具，迫使玩家做出取舍。
不可能买下所有东西，每次选择都是构筑方向的承诺。

## Player Fantasy

"代币永远不够用，每次购买都是一道选择题。"
玩家在"现在买还是攒钱等更好的"之间纠结，在"补强现有流派还是开辟新方向"之间权衡。
没有正确答案，只有不同的构筑路径。

## Detailed Design

### Core Rules

#### 代币

- 唯一货币，不可负债
- 余额为非负整数，初始为 0
- 每局开始时代币归零（无跨局继承）

#### 代币获取

击败敌人后获得代币奖励，数量由轮次决定：

| 轮次 | 奖励代币 |
|------|---------|
| 1 | 4 |
| 2 | 4 |
| 3 | 5 |
| 4 | 5 |
| 5 | 6 |
| 6 | 6 |
| 7 | 7 |
| 8 | 8 |
| **总计** | **45** |

从 economy.json 的 tokenRewards 数组按索引读取。

#### 代币花费

| 花费场景 | 操作 | 接口 |
|---------|------|------|
| 商店购买道具 | `spend(cost)` | 余额减少 cost，返回 true |
| 商店刷新 | `spend(1)` | 余额减少 1，商店重新抽取 |
| 购买失败（余额不足） | — | 返回 false，余额不变 |

#### 公开接口

```javascript
// 查询
economy.getBalance()           // 返回当前代币余额
economy.canAfford(cost)        // 返回 boolean

// 操作
economy.earn(amount)           // 获得代币（战斗胜利后调用）
economy.spend(cost)            // 花费代币（余额不足返回 false）
economy.reset()                // 新一局开始，余额归零
```

#### 典型经济流

```
战斗胜利 → earn(reward) → 进商店 → spend(购买) / spend(刷新) → 离开商店 → 下一轮
```

### States and Transitions

```
[初始(0)] --earn--> [有余额] --spend--> [余额减少] --...--> [本局结束]
                        |                        |
                        +---余额为0---> [破产状态（仍可继续战斗）]
```

- 破产不是游戏结束，只是暂时无法在商店消费
- 每轮战斗胜利后一定会获得代币（最小 4），不会永久破产

### Interactions with Other Systems

| 系统 | 交互方向 | 说明 |
|------|---------|------|
| 数据配置系统 | 读 | economy.json（奖励表、商店参数） |
| 游戏流程系统 | 写 | 战斗胜利后调用 earn()，新局调用 reset() |
| 商店系统 | 读 | 调用 canAfford() 和 spend() |
| UI系统 | 被读 | 显示当前代币余额 |

## Formulas

### 累计代币计算

```
cumulative(round) = Σ(i=1 to round) tokenRewards[i-1]
```

例：通关第3轮后累计 = 4 + 4 + 5 = 13 代币。

### 购买力估算

```
purchasingPower = floor(balance / avgItemCost)
```

用于 UI 提示（"你能买 N 个道具"），avgItemCost 按当前商店库存计算。

### 全局代币预算

全通关 45 代币，平均每个道具约 3.2 代币。预期可购买：

| 策略 | 购买数 | 每个平均花费 |
|------|--------|------------|
| 全买便宜货 | 10-12 个 | ~3.8 |
| 均衡购买 | 7-8 个 | ~5.6 |
| 攒钱买贵的 | 5-6 个 | ~7.5 |

刷新费用（每次 1 代币）会额外消耗预算，迫使玩家在"刷新找更好的"和"将就买现有的"之间决策。

## Edge Cases

| 情况 | 处理方式 |
|------|---------|
| 余额恰好等于道具价格 | 可以购买，购买后余额为 0 |
| 连续刷新多次 | 每次扣 1 代币，余额不足时拒绝刷新 |
| 战斗失败无代币获得 | 不调用 earn()，余额不变 |
| 代币达到极大值 | 理论最大值 45，无溢出风险 |
| 新局开始未 reset | 由游戏流程系统保证调用 reset() |
| earn(0) 或 spend(0) | earn(0) 无操作，spend(0) 返回 true（合法但不常见） |

|------|---------|---------|
| 数据配置系统 | 硬依赖 | economy.json（奖励表、商店参数） |

### 下游依赖（被依赖）

| 系统 | 依赖性质 | 读取内容 |
|------|---------|---------|
| 商店系统 | 硬依赖 | canAfford()、spend() |
| 出千能力系统 | 间接依赖 | 通过商店系统间接消费代币 |
| 游戏流程系统 | 硬依赖 | earn()、reset() |
| UI系统 | 软依赖 | getBalance() 显示余额 |

## Acceptance Criteria

| 编号 | 验收条件 | 验证方式 |
|------|---------|---------|
| AC-1 | 初始余额为 0 | 单元测试：验证新局开始余额 |
| AC-2 | earn 正确增加余额 | 单元测试：earn(5) 后余额 = 5 |
| AC-3 | spend 正确扣除余额 | 单元测试：余额 10，spend(3) 后余额 = 7 |
| AC-4 | 余额不足时 spend 返回 false 且余额不变 | 单元测试：余额 2，spend(3) 返回 false，余额仍为 2 |
| AC-5 | 全通关累计代币为 45 | 单元测试：模拟 8 轮 earn，验证累计 |
| AC-6 | canAfford 判断正确 | 单元测试：余额 5，canAfford(5)=true，canAfford(6)=false |
| AC-7 | reset 清零余额 | 单元测试：余额 30，reset 后为 0 |
| AC-8 | 代币不会为负 | 单元测试：任何操作后余额 ≥ 0 |
