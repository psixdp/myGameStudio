# 计分系统

> **Status**: Designed
> **Author**: user + agents
> **Last Updated**: 2026-04-03
> **Implements Pillar**: 支柱1（出千即爽感——分数爆炸）+ 支柱4（一目了然——分数清晰可读）

## Overview

计分系统负责三个核心任务：
1. **分类匹配**：给定骰子池，自动判断匹配哪种计分类别（7 种中选择最高优先级）
2. **分数计算**：按公式计算最终分数（基础分 + 加法加成 × 乘法倍率）
3. **胜负判定**：将最终分数与敌人目标分数比较

这是数值反馈的核心——所有出千能力的价值最终都体现在分数上。分类匹配算法需要处理
可变骰子数（3-7 个）× 7 种分类 × 优先级排序的组合复杂度，是整个游戏设计复杂度最高的系统。

## Player Fantasy

"看到一堆乱七八糟的骰子，然后用出千能力把它们改写成完美牌型，看着分数从 15 跳到 150。"
计分系统让这种爆炸式增长成为可能。玩家不需要手动选分类——系统自动选最优，
让玩家专注于"怎么改写骰子"这个决策。

分数的每一步增加都应该清晰可见（支柱4）：基础分多少、分类奖励多少、能力加成多少，
让玩家理解"为什么分数这么高"。

## Detailed Design

### Core Rules

#### 分类匹配算法

给定骰子池（3-7 颗骰子，可能含 1 颗临时骰子），按优先级从高到低检查每种分类，
返回第一个匹配的分类。匹配检查逻辑：

| 分类 | priority | minDice | 匹配条件 |
|------|----------|---------|---------|
| 豹子 | 1 | 3 | 所有骰子值相同 |
| 满堂红 | 2 | 5 | 存在 3 个相同 + 另 2 个相同 |
| 大顺 | 3 | 5 | 存在 5 个连续值 |
| 小顺 | 4 | 4 | 存在 4 个连续值 |
| 三条 | 5 | 3 | 存在 ≥3 个相同值 |
| 对子 | 6 | 2 | 存在 ≥2 个相同值 |
| 散牌 | 7 | 0 | 兜底，始终匹配 |

**匹配流程**：
1. 按 priority 升序排列所有分类
2. 跳过 minDice > 当前骰子数的分类
3. 跳过被敌人规则封锁的分类
4. 逐一检查匹配条件，返回第一个匹配的
5. 如果全部不匹配（理论上不会发生），返回散牌

#### 各分类匹配细节

**豹子（all_same）**：
- 条件：骰子池中所有骰子的值完全相同
- 例：[5,5,5] → 匹配；[5,5,4] → 不匹配

**满堂红（full_house）**：
- 条件：**恰好**存在一组值出现 ≥3 次，且另一组不同值出现 ≥2 次，且骰子池中只有这两组值
- 代码定义：
  ```javascript
  function isFullHouse(diceValues):
      freq = countFrequency(diceValues)   // {值: 出现次数}
      values = Object.keys(freq)
      if (values.length !== 2) return false          // 恰好两组值
      counts = Object.values(freq).sort()             // [小, 大]
      return counts[0] >= 2 && counts[1] >= 3         // 小组≥2, 大组≥3
  ```
- 例：[3,3,3,5,5] → 匹配；[3,3,3,3,5] → 不匹配（频率为 {4,1}，小组只有1次）
- **6-7 颗骰子时**：仍要求恰好两组值。如 [4,4,4,5,5,6] 不匹配（3组值），归入三条
- 满堂红匹配不放宽，这是三条后期价值更高的原因之一

**大顺（consecutive_5）**：
- 条件：去重后的骰子值中存在 5 个连续整数
- 可能序列：[1,2,3,4,5] 或 [2,3,4,5,6]
- 例：[1,2,3,4,5,6] → 匹配（6 颗也包含 5 连续）
- 顺子眼被动：允许间隔 1（如 [1,3,4,5] 算小顺），由出千能力系统修改匹配逻辑

**小顺（consecutive_4）**：
- 条件：去重后的骰子值中存在 4 个连续整数
- 可能序列：[1,2,3,4], [2,3,4,5], [3,4,5,6]
- 例：[2,3,4,5,1] → 匹配

**三条（same_value_3）**：
- 条件：存在某个值出现 ≥3 次
- 例：[4,4,4,2] → 匹配；[4,4,2] → 不匹配（只有 2 个）

**对子（same_value_2）**：
- 条件：存在某个值出现 ≥2 次
- 多组对子时取点数和较大的一组作为匹配依据
- 例：[5,5,2,2] → 以 5 的对子为准

**散牌（fallback）**：
- 始终匹配，无可匹敌条件

#### 分数计算

**完整公式**：

```
最终分数 = floor((分类基础分 + 加法加成总和) × 乘法倍率总和)
```

计算步骤：

1. **分类基础分** = 全部骰子点数之和 + 分类奖励分（或 × 分类倍率）
   - flat 类型：基础分 = sum(dice) + bonusValue
   - multiplier 类型：基础分 = sum(dice) × bonusValue

2. **敌人规则影响**（步骤 8）：如最低点归零，将最低骰子在计分中视为 0

3. **加法加成** = 所有被动能力的 flat 加成之和

4. **乘法倍率** = 所有被动能力的 multiplier 倍率之积
   - 例：贪欲 ×1.2 + 另一个 ×1.3 → 总倍率 = 1.2 × 1.3 = 1.56

5. **向下取整**：乘法计算后 floor 为整数

#### 分数计算示例

**示例 1**：骰子 [6,6,3]，无被动
- 匹配对子，flat 类型，bonusValue=0
- 基础分 = 6+6+3 + 0 = 15
- 加法加成 = 0，乘法倍率 = 1.0
- 最终分数 = floor(15 × 1.0) = **15**

**示例 2**：骰子 [6,6,6]，持有"牌型大师"和"贪欲"
- 匹配豹子，multiplier 类型，bonusValue=3
- 基础分 = 18 × 3 = 54
- 牌型大师（豹子 +10）：加法加成 = +10
- 贪欲（×1.2）：乘法倍率 = 1.2
- 最终分数 = floor((54 + 10) × 1.2) = floor(76.8) = **76**

**示例 3**：骰子 [5,5,5,5,2]，持有"连横术"
- 匹配三条（priority 5 > 对子的 6），same_value 值=5 出现 4 次
- 基础分 = 5+5+5+5+2 + 5 = 27
- 连横术：超出部分 = 4-3 = 1 颗，加成 = 1×3 = +3
- 加法加成 = 3，乘法倍率 = 1.0
- 最终分数 = floor((27 + 3) × 1.0) = **30**

### States and Transitions

计分系统在战斗中的调用流程：

```
[等待输入] --骰子就绪--> [分类匹配] --匹配完成--> [基础分计算]
      ↑                                              |
      |                                              v
[输出最终分数] <--取整完成-- [乘法倍率] <--加法加成-- [敌人规则影响]
```

计分是无状态的纯函数：给定输入（骰子池 + 被动列表 + 敌人规则），输出确定的分数。
不维护内部状态。

### Interactions with Other Systems

| 系统 | 交互方向 | 说明 |
|------|---------|------|
| 数据配置系统 | 读 | 读取 scoring-categories（分类定义、奖励参数） |
| 骰子系统 | 读 | 读取骰子池当前值进行匹配和求和 |
| 出千能力系统 | 读 | 读取当前被动能力列表（加法/乘法加成） |
| 敌人系统 | 读 | 读取当前敌人规则（封锁分类、最低点归零等） |
| 战斗系统 | 被调用 | 战斗系统在结算阶段调用计分系统 |
| UI系统 | 被读 | UI 读取分数明细（基础分、加成、倍率）做展示 |

## Formulas

### 分类基础分

**关键区分：分类匹配用原始骰子值，计分求和用敌人规则修改后的值。**

```
// === 分类匹配阶段（使用原始骰子值）===
rawDiceValues = dicePool.getValues()
matchedCategory = findBestCategory(rawDiceValues, blockedCategories)

// === 计分阶段（使用敌人规则修改后的值）===
modifiedDiceValues = applyScoringRules(rawDiceValues, enemyRules)  // 如最低点归零
diceSum = sum(modifiedDiceValues)

if (bonusType == "flat"):
    categoryBase = diceSum + bonusValue
elif (bonusType == "multiplier"):
    categoryBase = diceSum * bonusValue
```

这意味着：敌人规则"最低点归零"不会影响分类匹配判断（如 [1,5,5] 仍匹配对子），
但会影响基础分计算（sum 时最低骰子计为 0）。

### 加法加成总和

```
flatBonus = Σ (passive.flatBonus where passive.category == matched OR passive.category == "all")
```

加法加成的触发条件：
- 牌型大师：仅当匹配满堂红、豹子或三条时生效（+20）
- 连横术：当匹配三条/对子/满堂红时，超出部分的加成

### 连横术加成计算

```
// 豹子特殊处理：所有骰子一致，无"超出"概念
if (matchedCategory.id == "yahtzee"):
    linkBonus = 0
else:
    matchedCount = max_frequency(diceValues)   // 匹配值出现的次数
    requiredCount = category.matchCount        // 分类最低要求
    excessCount = max(0, matchedCount - requiredCount)
    linkBonus = excessCount * 5
```

豹子要求所有骰子一致，不存在"超出最低要求"的骰子，因此连横术对豹子加成固定为 0。
牌型大师作为豹子的专属补偿加成（+10）。

### 乘法倍率总和

```
totalMultiplier = Π (passive.multiplier)
```

所有乘法加成用积叠加。贪欲 ×1.2 + 另一个 ×1.5 = 1.8。

### 最终分数

```
finalScore = floor((categoryBase + flatBonusTotal) * totalMultiplier)
```

### 连续值检测（顺子）

```
function hasConsecutive(values, length):
    sorted = unique(values).sort()
    count = 1
    for i from 1 to sorted.length:
        if sorted[i] == sorted[i-1] + 1:
            count++
            if count >= length: return true
        else:
            count = 1
    return false
```

### 对子匹配值选择

```
function bestPairValue(diceValues):
    groups = groupByValue(diceValues)
    candidates = groups.filter(g => g.count >= 2)
    return candidates.maxBy(g => g.value)   // 取点数最高的对子
```

多组对子时选择点数最高的值作为匹配依据。
例：[5,5,2,2] → 选 5 的对子；[6,6,3,3] → 选 6 的对子。

## Edge Cases

| 情况 | 处理方式 |
|------|---------|
| 所有骰子被最低点归零（敌人规则） | sum(dice) 中最低骰子贡献 0，分类匹配仍用原始值判断 |
| 豹子 × 连横术 | 豹子要求所有骰子一致，无超出部分，连横术对豹子加成 = 0 |
| 满堂红 + 6颗骰子 | 严格要求 3+2 结构，[4,4,4,5,5,6] 不匹配满堂红（有第 3 组值） |
| 满堂红 + 7颗骰子 | 同上，严格要求恰好两组值，频率 {3,2}，多余骰子破坏匹配 |
| 大顺 + 6颗骰子 | [1,2,3,4,5,6] 包含 5 连续，匹配大顺 |
| 顺子眼 + 小顺 | 由出千能力系统标记骰子池状态，计分系统检测到标记后放宽连续条件 |
| 散牌 + 所有分类被封 | 散牌（fallback）不可被封，始终兜底 |
| 加法加成为负数 | 允许，但最终分数最低为 0（不允许负分） |
| 乘法倍率 < 1 | 允许（如果有减速类敌人规则），但不低于 0.1 |
| 分数为 0 | floor 后为 0，与目标分数比较必然失败（除非目标也为 0） |
| 临时骰子参与计分 | 临时骰子值正常参与 sum 和分类匹配 |
| 多组对子出现 | 取点数和较大的一组；[5,5,2,2] 以 5 为准 |

## Acceptance Criteria

| 编号 | 验收条件 | 验证方式 |
|------|---------|---------|
| AC-1 | 3 颗骰子的所有分类匹配正确 | 单元测试：穷举 6³=216 种组合，验证分类结果 |
| AC-2 | 4 颗骰子的所有分类匹配正确 | 单元测试：穷举 6⁴=1296 种组合 |
| AC-3 | 5 颗骰子的所有分类匹配正确 | 单元测试：抽样验证（6⁵=7776 太大，按分类等价类抽样） |
| AC-4 | 基础分计算与手算一致 | 单元测试：给定骰子+分类，验证分数 |
| AC-5 | 加法加成正确叠加 | 单元测试：多个 flat 加成求和 |
| AC-6 | 乘法倍率正确叠加（积） | 单元测试：多个 multiplier 乘积 |
