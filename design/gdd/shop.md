# 商店系统

> **Status**: Designed
> **Author**: user + agents
> **Last Updated**: 2026-04-03
> **Implements Pillar**: 支柱3（联动 > 单体——购买选择决定构筑方向）

## Overview

商店系统管理每轮战斗后的购买流程。从能力池中随机抽取道具展示给玩家，
玩家用代币购买。核心决策是"买什么"和"刷新还是将就"。

商店是构筑方向的决策点——每次购买都是对一种流派的承诺。
有限的代币迫使玩家取舍，不可能买下所有东西。

## Player Fantasy

"每次进商店都像在赌场里挑道具——这件便宜但一般，那件贵但能联动我现有的能力。"
刷新的花费制造了"要不要再看看"的纠结，而骰子扩展在前几轮的加权出现
引导了一个自然的早期决策："先扩展骰子解锁更多分类"。

## Detailed Design

### Core Rules

#### 商店开放时机

每轮战斗胜利后开放商店，购买完成后（或跳过商店后）进入下一轮。

#### 道具抽取

每次进商店从 abilities.json 中随机抽取 3 个道具展示：
- 使用 shop 随机流进行抽取
- 排除已拥有的被动能力（不重复出售）
- 骰子扩展在骰子池已满（7颗）时不再出现
- 骰子扩展在前 3 轮出现概率加权（bonusWeight=2.0）

#### 加权抽取算法

```
function weightedShopPick(abilityPool, count, round):
    candidates = abilityPool.filter(a => !isOwned(a))

    // 骰子池已满时排除骰子扩展
    if (dicePool.isFull()):
        candidates = candidates.filter(a => a.type != 'dice_expansion')

    // 构建权重
    weighted = candidates.map(a => {
        weight = 1.0
        if (a.type == 'dice_expansion' && round in bonusRounds):
            weight *= bonusWeight   // 前三轮加权
        return { item: a, weight: weight }
    })

    return shopRng.weightedPick(weighted, count, noRepeat: true)
```

#### 购买流程

```
[展示道具] → 玩家选择购买/刷新/离开
    ├── 购买: 检查余额 → 扣款 → 能力加入玩家库存 → 刷新空位
    ├── 刷新: 检查余额 → 扣1代币 → 重新抽取3个道具
    └── 离开: 结束商店阶段
```

#### 购买规则

- 余额 ≥ 道具价格才可购买
- 购买后道具从展示列表移除，空位不自动补充
- 被动能力不重复出售（已拥有的不出现）
- 消耗品可重复购买（无拥有上限，仅受代币限制）
- 骰子扩展购买后立即生效（调用骰子系统 addPermanentDie）

#### 刷新规则

- 刷新费用：1 代币（从 economy.json 读取）
- 可多次刷新
- 刷新后展示全新的 3 个道具（重新抽取）
- 刷新不保留之前未购买的道具

#### 公开接口

```javascript
// 生命周期
shop.open(round, dicePool)           // 打开商店，抽取初始道具
shop.close()                          // 关闭商店

// 查询
shop.getDisplayItems()                // 返回当前展示的道具列表
shop.canBuy(itemIndex)                // 余额是否足够
shop.canRefresh()                     // 余额是否够刷新

// 操作
shop.buy(itemIndex)                   // 购买，返回 true/false
shop.refresh()                        // 刷新道具，返回 true/false
```

### States and Transitions

```
[关闭] --open()--> [展示道具] ←--refresh()── [展示道具]
                      |                           |
               buy()  +──buy()──→ [购买成功] ──→ [更新展示]
                      |
                      +--close()→ [关闭]
```

购买后展示列表中对应位置变为空（或移除），玩家可继续购买剩余道具或离开。

### Interactions with Other Systems

| 系统 | 交互方向 | 说明 |
|------|---------|------|
| 数据配置系统 | 读 | abilities.json（能力池）、economy.json（商店参数） |
| 随机数系统 | 读 | shop 流（道具抽取） |
| 经济系统 | 写 | spend()（购买和刷新扣款） |
| 出千能力系统 | 写 | 购买后能力加入玩家库存 |

| UI系统 | 被读 | 展示道具列表、价格、余额 |

## Formulas

### 道具抽取权重

```
baseWeight = 1.0
if (type == 'dice_expansion' && round ∈ [1,2,3]):
    finalWeight = baseWeight × 2.0
else:
    finalWeight = baseWeight
```

### 骰子扩展出现概率

普通轮次（4-8轮）骰子扩展概率 = 扩展数量 / 候选池大小
前三轮加权后概率 ≈ 普通 × 2.0

## Edge Cases

| 情况 | 处理方式 |
|------|---------|
| 候选池不足3个道具 | 展示所有可用的（可能少于3个） |
| 候选池为空 | 商店展示"已售罄"，玩家只能离开 |
| 已拥有所有被动 | 被动不再出现，只展示消耗品和骰子扩展 |
| 骰子池已满(7) | 骰子扩展从候选池移除 |
| 余额为0时进商店 | 可浏览但无法购买或刷新，只能离开 |
| 购买后展示列表为空 | 自动提示"已购空"，可离开 |
| 刷新后候选池不足3个 | 有几个展示几个 |
| 同时有消耗品和骰子扩展要展示 | 按权重随机，不保证每种类型都有 |
| 千王骰（初始固定为6） | 购买时调用 addPermanentDie(6)，之后每轮正常投掷 |

## Acceptance Criteria

| 编号 | 验收条件 | 验证方式 |
|------|---------|---------|
| AC-1 | 打开商店正确抽取3个道具 | 单元测试：验证展示列表长度 ≤ 3 |
| AC-2 | 不展示已拥有的被动 | 单元测试：拥有铅骰后刷新不含铅骰 |
| AC-3 | 骰子池满时不展示骰子扩展 | 单元测试：7骰子时展示列表无扩展 |
| AC-4 | 购买成功扣款并加入库存 | 集成测试：余额减少，能力列表增加 |
| AC-5 | 余额不足时购买失败 | 单元测试：余额<价格时 buy() 返回 false |
| AC-6 | 刷新扣1代币并展示新道具 | 单元测试：余额-1，展示列表更新 |
| AC-7 | 余额不足时无法刷新 | 单元测试：余额=0 时 canRefresh()=false |
| AC-8 | 前三轮骰子扩展概率更高 | 统计测试：1000次抽取，前三轮扩展出现率 > 后续轮次 |
| AC-9 | 同一种子道具列表一致 | 单元测试：固定种子，验证两局商店内容相同 |
| AC-10 | 千王骰购买时初始值为6 | 集成测试：购买后骰子池新增骰子值为6 |
